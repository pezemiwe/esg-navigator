## 1. Duplicate assessment on every "New Assessment" click — fixed

**Files:** `src/services/projectApi.ts`, `src/store/sustainabilityStore.ts`, `server/src/routes/projects.ts`

**What:** `apiCreateProject(userId, payload)` now requires an `id` field in `payload` and sends it
in the POST body. `createNewProject()` in the store passes its locally-generated UUID through.
`POST /api/projects` on the server now does `prisma.assessmentProject.create({ data: { ...(body.id
&& { id: body.id }), ... } })` — uses the client's id when supplied, generates its own otherwise
(keeps the route backward-compatible for any other caller).

**Why:** `createNewProject()` generates a local UUID and immediately sets it as `activeProjectId`,
but the old `apiCreateProject` call never sent that id — the server minted its own `cuid()`,
creating an orphaned row nobody ever referenced again. The very next `saveCurrentProject()` (fired
by the layout's route-change effect) then PUT-upserted a *second* row under the local UUID. Net
result: one click, two server rows, confirmed live via direct DB queries during Playwright testing
(count jumped by more than 1 per click; UI-reported assessment counts also diverged from actual DB
row counts, meaning some client-created projects weren't reaching the server at all).

**Frontend impact:** None visible — same button, same navigation. `apiCreateProject`'s signature
changed (now takes `{ id, groupName, isGroupAssessment }` instead of `{ groupName,
isGroupAssessment }`); grep for other callers before touching this function again — there was only
one caller (`createNewProject`) at the time of this fix.

**Verified:** Live test hitting `GET /api/projects` before/after a create click — server count now
goes up by exactly 1, and the new row's id matches the local `activeProjectId`.

## 2. Deleted assessment reappearing after a page refresh — fixed

**File:** `src/store/sustainabilityStore.ts` — `deleteProject`

**What:** `deleteProject` still removes the card from local state immediately (optimistic, same as
before), but now `await`s `apiDeleteProject` instead of firing it and forgetting. On failure, it
rolls back — re-adds the removed project to `assessmentProjects` and, if it was the active project,
restores the full pre-delete state slice (`activeProjectId`, `governanceAssessment`, `valueChain`,
`srroItems`, `phase4Entries`, `phase5Items`, entity fields, approval state) — and shows
`toast.error("Couldn't delete the assessment on the server — it has been restored.")` via the
existing `@/components/ui` toast store.

**Why:** The delete itself was never the slow part — the card vanished from the UI instantly. The
bug was that a failed or slow DELETE request had no consequence: `apiDeleteProject` only did
`console.warn` on error. `syncFromServer()` (runs on every `SustainabilityDashboard` mount) merges
back any server project missing from local state — so if the DELETE hadn't actually landed
server-side, refreshing the page would silently bring the "deleted" item back with no explanation.

**Frontend impact:** `deleteProject`'s call signature is unchanged (`(id: string) => void`), so no
caller needs to change. The only visible difference is a toast on the rare failure path — nothing
changes on the success path.

**Verified:** Live test — create then delete via the UI, checking `GET /api/projects` after each
step. Server count returns exactly to baseline after delete, with the deleted id absent.

## 3. Entity Structure (parent/subsidiary) — design spec written, NOT implemented

**File:** `docs/entity-structure-parent-child-spec.md` (new)

**What:** Wrote a spec for how the Governance Assessment "Entity Structure" section should let a
user pick an existing entity as a parent when adding a new one, and add a "promote to group" action
for retroactively wrapping already-added standalone entities under a new parent. No code changed.

**Why it's needed:** Current UI (`GovernanceAssessment.tsx` ~lines 439-530) only offers a binary
Standalone/Group radio; "Group Name" is a free-text label, not a real entity record; every
subsidiary added via `addEntity()` hard-codes `parentId: null`. There's no way to select an
existing entity as parent, and no way to add a parent after subsidiaries already exist.

**Good news for scoping this:** no backend/schema change needed. `Entity.parentId` is already a
self-relation in `schema.prisma` supporting arbitrary nesting, `sectorId`/`subSector` already live
on `Entity` (not the project) — correct, since a diversified group's subsidiaries can span
different SASB industries. The frontend type `AssociatedEntity` already has `parentId: string |
null` and `EntityRelationshipType` already includes `"Parent"` as a valid value; it's just never
offered in the UI's entity-type `<select>`. Full detail, including the two new store actions this
would need (`reassignEntityParent`, `promoteToGroup`) and the cycle-guard requirement, is in the
spec doc.

**Status:** Deliberately held — project owner asked to hold off on frontend changes here pending
review of the spec.
