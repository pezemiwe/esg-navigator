Audience: primary developer. Covers all changes since the initial backend sprint.
Branch: `backend-dev`. No commits yet — all changes unstaged.

## 1. LLM client: env var read timing fix

**File:** `server/src/llm/index.ts`

**What:** Moved `const LLM_CLIENT` and `const ANTHROPIC_MODEL` from module-level constants into
the body of `callLLM()`.

**Why:** tsx v4.x uses native ESM, which hoists `import` statements before any top-level code.
`dotenv.config({ path: '../.env' })` in `server/src/index.ts` runs *after* all imported modules
have been evaluated. So when `LLM_CLIENT` was a module-level constant it was always evaluated as
`undefined` (no env yet), defaulting to `'anthropic'` — but because the previous server process
had been started without env vars it was silently returning mock responses at ~0.15 s.

Moving the read inside the function means it executes at *call time*, after dotenv has run.
Verified: real Anthropic calls now take ~16 s and return 10–15 sector-specific items.

## 2. Database path: wrong `DATABASE_URL` in `.env.example`

**File:** `server/.env.example`

**What:** Changed `DATABASE_URL` from `"file:./prisma/dev.db"` to `"file:./dev.db"`.

**Why:** Prisma resolves SQLite file paths **relative to `schema.prisma`**, not relative to the
server process CWD. `schema.prisma` lives in `server/prisma/`, so `file:./dev.db` resolves to
`server/prisma/dev.db` — which is where migrations create the database. The earlier path
(`file:./prisma/dev.db`) resolved to `server/prisma/prisma/dev.db` (a directory that doesn't
exist), causing "Error code 14: Unable to open the database file" on every API call.

Note: `server/.env` is gitignored. The correct value to copy is `file:./dev.db`.

## 3. Single-entity save path: top-level store data now reaches the DB

**File:** `src/store/sustainabilityStore.ts` — `saveCurrentProject` fire-and-forget block

**What:** When `assessmentEntities` is empty (the normal case for a non-group assessment),
`saveCurrentProject` now synthesises a single entity payload from the top-level store fields
(`governanceAssessment`, `valueChain`, `srroItems`, `phase4Entries`, `phase5Items`) rather than
mapping an empty array and sending nothing.

**Why:** The multi-entity code path iterates `state.assessmentEntities`. For single-entity
assessments that array is always empty — all phase data lives at top level, not in
`entitySnapshots`. The API was receiving `entities: []` on every save, so the DB for a real
assessment always had 0 entities and 0 SRRO items. This was invisible because the fire-and-forget
pattern suppresses errors.

**Frontend impact:** The synthesised entity uses `state.activeEntityId || "parent"` as its ID and
`governanceAssessment.clientName` as its name — making its DB record meaningful, not an anonymous
placeholder.

## 4. PUT route: upsert instead of update; removed ownership pre-check

**File:** `server/src/routes/projects.ts`

**What:**
1. Removed the `findFirst({ where: { id, userId } })` ownership check that returned 404 before any
   write happened.
2. Changed `prisma.assessmentProject.update(...)` to `prisma.assessmentProject.upsert(...)` with a
   `create` branch that seeds the project record from the request body if it doesn't yet exist.

**Why:** When a user creates an assessment in the UI, `createNewProject` writes a UUID to
localStorage and calls `POST /api/projects` to create the DB record. But if the server was down
(or the DB path was wrong, as above) that POST silently failed. The next save attempt then hit a
`PUT /api/projects/:id` for a project ID that didn't exist in the DB — the `findFirst` guard
returned 404, the fire-and-forget swallowed the error, and nothing was ever persisted.

The upsert means the first successful PUT creates the record. This is safe: the client owns the
UUID; the server trusts `x-user-id` (mock auth by design per CLAUDE.md).

## 5. Demo data: auto-seed removed, manual "Load sample data" added

**File:** `src/features/sustainability/pages/SRRORegister.tsx`

**What:** Removed the `useEffect` that called `setSrroItems(PHASE3_INITIAL_DATA)` on every mount
when `srroItems.length === 0`. Added a "Load sample data" button that appears only when the
register is empty and unlocked, which calls `setSrroItems(PHASE3_INITIAL_DATA)` on demand.
`PHASE3_INITIAL_DATA` and `src/config/phase3InitialData.ts` are retained.

**Why the auto-seed was wrong:**
- It fired on *every* mount of the SRRO Register page, not just on first creation. Any assessment
  with an empty register — including a brand-new real assessment the user just created — would
  immediately be populated with 85 hardcoded items belonging to a fictitious Nigerian insurer
  ("HGI").
- Because Phases 4 and 5 derive their row lists from `srroItems.filter(i => i.includeInFinalList === "Yes")`,
  all three phases showed "demo" content that had nothing to do with the real client. Every
  assessment looked like HGI's assessment.
- The seed also fired after a user *deliberately* cleared the register, making it impossible to
  work with an intentionally empty Phase 3.

**Frontend justification:** This is a behaviour correction, not a UX regression. The sample data
is preserved and opt-in. The button is only visible when the register is empty, so it does not
clutter assessments that have been populated.

## 6. AI generate: structured Value-Chain data forwarded to the prompt

**Files:** `src/services/srroApi.ts`, `src/features/sustainability/pages/SRRORegister.tsx`,
`server/src/routes/srro.ts`

**What:** Extended the `POST /api/srro/generate` request payload to include:
- `businessModelContext` — business model description, key products/services, key markets/regions
- `activities` — value-chain activity register (stage, activity, description, inputs, outputs)
- `resources` — resources & relationships register (vendor, stage, capital type, etc.)

Also fixed sector/subSector sourcing: both are now read from `governanceAssessment` first, falling
back to the `scenarioStore` values only when governance fields are blank.

The server `buildUserMessage` function renders these structured fields into the prompt between the
entity header and the questionnaire responses, so the LLM receives the full Phase-2 picture.

**Why:**
- The intended design is "Phase 2 feeds Phase 3". Before this change, only the flat questionnaire
  key/value pairs reached the model. The richer structured data (activities register, resources
  register, business model) was collected in the UI but silently dropped at the API boundary —
  the `srroApi.ts` payload type had no fields for it.
- `subSector` was previously hardcoded to `sector.subSectors[0]` (the first sub-sector in the
  sector config catalogue), meaning a sector with multiple sub-industries always sent the wrong
  sub-sector. Governance now takes precedence.
- `sector` was sourced from `scenarioStore`, a separate module from the sustainability assessment.
  Using `governanceAssessment.sector` makes the generate call self-consistent with the entity being
  assessed.

**Frontend justification:** The payload extension is additive (all new fields are optional on the
server). The existing questionnaire-response path is unchanged. No UI changes to the generate
button or error handling.

## 7. Syntax fix: nullish coalescing operator precedence

**File:** `src/features/sustainability/pages/SRRORegister.tsx:183`

**What:** Added parens around the `??` fallback chain in `handleGenerateWithAI`:
`sector: governanceAssessment.sector || (sectorFallback?.name ?? selectedSectorId ?? "Unknown sector")`

**Why:** Babel/Vite's parser rejects `expr || a ?? b` without explicit parens (they flag it as
ambiguous operator precedence). The expression was syntactically valid TypeScript but caused a
Vite internal server error (500) on every page load of the SRRO Register, preventing the
component from rendering at all. The fix is a one-character change.

## How to run

```bash
# From project root
npm run dev:all          # Vite :3000 + Hono :8787

# Verify API is up
curl localhost:8787/api/health
```

Environment (copy `server/.env.example` to `server/.env`, then fill):
```
DATABASE_URL="file:./dev.db"
LLM_CLIENT=anthropic
ANTHROPIC_API_KEY=<your key>
ANTHROPIC_MODEL=claude-haiku-4-5-20251001
```

Root `.env` (project root, not `server/`):
```
LLM_CLIENT=anthropic
ANTHROPIC_API_KEY=<your key>
```

The server reads from root `.env` via `dotenv.config({ path: '../.env' })` in `server/src/index.ts`.
